# Technical Debt & Risk Register

> Cumulative register across all layers, sorted by severity descending (Critical -> High ->
> Medium -> Low). IDs are sequential and never reset.

## Technical Debt & Risk Register

| ID | Risk / Debt Item | Category | Affected Component(s) | Severity | Evidence | Recommended Action |
|---|---|---|---|---|---|---|
| TD-01 | PostgreSQL credentials (`postgres`/`Clarium123`) are hardcoded in plaintext and committed to source control for both `CatalogConnection` and `IdentityConnection` | Security Vulnerability | Web, PublicApi | **Critical** | `ConnectionStrings:CatalogConnection`/`IdentityConnection` in src/Web/appsettings.json and src/PublicApi/appsettings.json, value `Host=localhost;Port=5432;...Username=postgres;Password=Clarium123` | Remove credentials from source-controlled appsettings; rotate the Postgres password; load via .NET User Secrets (local) and a managed secret store (Key Vault/env var injection) for any shared/Docker environment |
| TD-02 | SQL Server / Azure SQL Edge `sa` password (`@someThingComplicated1234`) is hardcoded in plaintext in `docker-compose.yml` and repeated in both services' `appsettings.Docker.json` connection strings | Security Vulnerability | sqlserver (compose), Web, PublicApi | **Critical** | `SA_PASSWORD=@someThingComplicated1234` (docker-compose.yml); `Password=@someThingComplicated1234` in `CatalogConnection`/`IdentityConnection` (src/Web/appsettings.Docker.json, src/PublicApi/appsettings.Docker.json) | Rotate the `sa` password; inject via Docker secrets or environment variable substitution from a non-committed `.env` file (add to `.gitignore`); never reuse the same password across multiple connection strings |
| TD-03 | `mcr.microsoft.com/azure-sql-edge` is referenced with no version tag (resolves to `latest`), and Azure SQL Edge itself has been retired by Microsoft (end of support 2025-09-30) | EOL Technology / Configuration Risk | sqlserver (compose) | **Critical** | `image: mcr.microsoft.com/azure-sql-edge` (docker-compose.yml, no tag) - DISCREPANCY vs Agent 1's LOW (Agent 1 flagged missing tag only; this pass adds the EOL status) | Migrate the local/Docker database off Azure SQL Edge to a supported engine (e.g. `mcr.microsoft.com/mssql/server` with a pinned LTS tag, or a pinned PostgreSQL image matching the default-environment provider) before the `latest` tag becomes unresolvable or behaviourally inconsistent |
| TD-04 | Two RDBMS providers (Npgsql for default/local, SqlServer for Docker) are configured against identical connection-string keys (`CatalogConnection`, `IdentityConnection`) for the same shared EF Core model, with no documented rationale and no migration files to verify schema parity between the two engines | Architecture Anti-pattern / Configuration Risk | Web, PublicApi, Infrastructure | High | src/Web/appsettings.json vs appsettings.Docker.json; src/PublicApi/appsettings.json vs appsettings.Docker.json; empty `Data\Migrations\` folder in Infrastructure.csproj - carried/escalated from Agent 1's flagged DISCREPANCY | Confirm with the team whether dual-provider support is intentional (e.g. Postgres for local dev cost savings, SQL Server for Azure deployment parity); if intentional, add migration files for both providers and a CI job that runs tests against both; if unintentional drift, standardise on one provider |
| TD-05 | No retry, circuit-breaker, or timeout policy exists for any database connection (Postgres or SQL Server) or for inter-service HTTP calls (BlazorAdmin -> PublicApi, BlazorAdmin -> Web) | Scalability Constraint / Operational Risk | Web, PublicApi, BlazorAdmin, Infrastructure | High | No Polly/`Microsoft.Extensions.Http.Resilience`/`EnableRetryOnFailure` found in any `.csproj`; no timeout values in any connection string or appsettings (NFR-08, NFR-23, Pattern Coverage Gaps) | Add EF Core `EnableRetryOnFailure()` with explicit max-retry-count/delay for both providers; add `HttpClient` timeout + retry policy for BlazorAdmin's calls to PublicApi/Web |
| TD-06 | CI/CD pipeline (`dotnetcore.yml`) performs only `dotnet build` + `dotnet test` with no SAST, dependency vulnerability scan, container/image scan, secret scan, or IaC scan - the two Critical hardcoded-secret findings (TD-01, TD-02) would pass undetected | CI-CD Risk | All (pipeline-wide) | High | Full step-by-step read of .github/workflows/dotnetcore.yml and richnav.yml - no matching tool invocations found for any of: SAST, dependency scan, container scan, secret scan, IaC scan (see Operational Architecture Assessment) | Add, at minimum: a secret-scanning step (e.g. gitleaks) and a dependency-vulnerability step (e.g. `dotnet list package --vulnerable` or equivalent) to the existing `dotnetcore.yml` workflow, gated on every push/PR |
| TD-07 | No automated deployment step exists in any GitHub Actions workflow, despite `infra/main.parameters.json` and `infra/abbreviations.json` indicating an intended Azure (azd) deployment target | CI-CD Risk / Dependency Coupling | Infra deployment pipeline (absent) | High | .github/workflows/dotnetcore.yml and richnav.yml contain no `azd deploy`, `az webapp deploy`, `terraform apply`, or similar; `infra/` contains parameters only, no `main.bicep`/resource template | Either add a CD workflow (e.g. `azd pipeline config` generated workflow) that consumes `infra/main.parameters.json`, or document that deployment is performed out-of-band (manual `azd up`) so the gap is intentional and understood |
| TD-08 | No confirmed JWT issuer/audience/signing-key configuration exists for PublicApi or Web despite both referencing `Microsoft.AspNetCore.Authentication.JwtBearer` and `System.IdentityModel.Tokens.Jwt` | Security Vulnerability (unverified control) | Web, PublicApi, Infrastructure | High | No `Jwt`/`Authentication`/`Authority`/`Audience` keys in any of the 6 reviewed appsettings files (AP-10) | Confirm in `Program.cs`/user-secrets whether JWT bearer auth is actually configured for PublicApi; if missing, either complete the configuration (if PublicApi endpoints should be token-protected) or remove the unused packages |
| TD-09 | No CORS policy configuration found despite BlazorAdmin's `apiBase`/`webBase` pointing at different origins/ports than PublicApi/Web in every declared environment (e.g. `localhost:44315` vs `localhost:5099`; `host.docker.internal:5106` vs `localhost:5200`) | Security Vulnerability | PublicApi, Web | High | src/BlazorAdmin/wwwroot/appsettings*.json (cross-origin `baseUrls`); no `Cors`/`AllowedOrigins` configuration found in any Web/PublicApi appsettings | Confirm CORS policy exists in `Program.cs`; if relying on a wildcard origin policy, tighten to the specific BlazorAdmin origins per environment |
| TD-10 | Shared database schema (Catalog + Identity) is accessed directly via EF Core by two independently deployable services (Web, PublicApi) with no API boundary between them | Dependency Coupling (shared-database antipattern) | Web, PublicApi, Infrastructure | Medium | Infrastructure.csproj referenced by both Web.csproj and PublicApi.csproj; identical `CatalogConnection`/`IdentityConnection` keys in both services' appsettings (Coupling Hotspots) | If Web and PublicApi are meant to evolve independently, consider routing one through the other's API rather than sharing direct DB access; at minimum, document the shared-schema contract and require coordinated migration review |
| TD-11 | `Microsoft.EntityFrameworkCore.InMemory` is referenced directly by `Web.csproj` and `PublicApi.csproj` (production projects), not only by the four test projects | Configuration Risk | Web, PublicApi | Medium | src/Web/Web.csproj, src/PublicApi/PublicApi.csproj `PackageReference Include="Microsoft.EntityFrameworkCore.InMemory"`; SHARED COMPONENT flagged by Agent 1 across Application and Test layers | Confirm the runtime switch (likely an `UseOnlyInMemoryDatabase` config flag) cannot be accidentally enabled in a real deployment (e.g. via a stray `appsettings.*.json` or environment variable); consider isolating the InMemory provider to test-only build configurations |
| TD-12 | `actions/checkout@v2` and `actions/setup-dotnet@v1` are used in both workflow files; these major versions are multiple releases behind current (`@v4`) | CI-CD Risk | .github/workflows/dotnetcore.yml, .github/workflows/richnav.yml | Medium | Direct read of both workflow files | Bump to `actions/checkout@v4` and `actions/setup-dotnet@v4` (or latest) to pick up Node 20 runtime support and security fixes in the actions themselves |
| TD-13 | `include-prerelease: true` is set for `actions/setup-dotnet` in the main CI build (`dotnetcore.yml`), meaning the exact .NET 8 SDK build used to compile/test "Release" artifacts is not pinned and may include preview/RC SDK releases | Configuration Risk | .github/workflows/dotnetcore.yml, .github/workflows/richnav.yml | Medium | `dotnet-version: '8.0.x'`, `include-prerelease: true` (dotnetcore.yml) | Remove `include-prerelease: true` for the main build/test workflow (acceptable for richnav.yml's code-indexing-only purpose, but not for the artifact-producing build) to ensure reproducible Release builds |
| TD-14 | Neither `src/Web/Dockerfile` nor `src/PublicApi/Dockerfile` declares a `HEALTHCHECK` instruction; `docker-compose.yml`'s `depends_on` for `sqlserver` has no `condition: service_healthy` | Operational Risk | eshopwebmvc, eshoppublicapi, sqlserver (compose) | Medium | Direct read of both Dockerfiles (no `HEALTHCHECK`); docker-compose.yml `depends_on` is a bare list, not a map with health conditions (NFR-26, NFR-28) | Add `HEALTHCHECK` instructions to both Dockerfiles (e.g. curl against a `/health` endpoint - which itself does not yet exist, see Pattern Coverage Gaps); add `depends_on.sqlserver.condition: service_healthy` once a SQL Server health check is available |
| TD-15 | `PublicApi.Dockerfile` declares `EXPOSE 80` and `EXPOSE 443`, but the actual runtime binding (via `ASPNETCORE_URLS=http://+:8080` in docker-compose.override.yml) is port 8080 - the Dockerfile's `EXPOSE` directives do not reflect the real listening port | Configuration Risk | eshoppublicapi | Low | src/PublicApi/Dockerfile (`EXPOSE 80`, `EXPOSE 443`) vs docker-compose.override.yml (`ASPNETCORE_URLS=http://+:8080`, `5200:8080`) - carried from Agent 1's flagged LOW | Update `EXPOSE` directives in src/PublicApi/Dockerfile to `8080` (and add `8443` only if HTTPS is actually terminated in-container) to keep the Dockerfile self-documenting |

| TD-17 | SQL Server (Azure SQL Edge) container port 1433 is published directly to the host (`1433:1433`) in `docker-compose.yml`, increasing the database's network attack surface beyond what the two application services require | Operational Risk / Attack Surface | sqlserver (compose) | Medium | docker-compose.yml `ports: - "1433:1433"` | Remove the host port mapping for `sqlserver` unless host-level DB tooling access is required for local development; rely on the Docker network for inter-container connectivity |
