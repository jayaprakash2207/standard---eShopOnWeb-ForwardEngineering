# 14 — Non-Functional Requirements (NFR) Specification

> **Single source of truth:** `ENTERPRISE_KNOWLEDGE_GRAPH.json`. Every requirement in this document traces to graph node ids (`TECH-CUR` / `TECH-INF` / `TECH-SEC` current-stack, infrastructure and security nodes; `APP-SVC` / `APP-DEP` services and dependencies; `DATA-ENT` / `DATA-REPO` data nodes) and to the evidence-anchored `ta-outputs/nfr-registry.md` (NFR-01..NFR-28) already cited inside those nodes. No technologies, infrastructure, or controls have been invented.
>
> **Measured vs recommended.** The legacy codebase contains **almost no declared NFR values**: the `nfr-registry` (NFR-01..28) is dominated by `**None declared**` rows where framework defaults apply (connection pooling, timeouts, retries, rate limits, cache TTL, health checks, resource limits — NFR-07, NFR-08, NFR-23..NFR-28). Where the graph/legacy has **no measured value**, this document supplies a **clearly-labelled recommended baseline** — a *neutral engineering target tied to the target architecture*, not a discovered fact — and ties it to a Forward-Engineering Assumption (`ASMP-FE-###`).
>
> **Technology neutrality.** `target_stack` is **empty (0 nodes)** in the graph. Where the legacy stack is named it is labelled **Current (legacy)** (e.g. .NET 8 / `TECH-CUR-001`, Azure SQL Edge / `TECH-CUR-020`, PostgreSQL / `TECH-CUR-021`, Docker Compose / `TECH-INF-004`). Target runtimes, datastores, and platforms are presented as **neutral options ("not in legacy evidence")** spanning the mandated set (Java Spring Boot, ASP.NET Core, Node.js, Python; React/Angular/Vue; PostgreSQL/SQL Server/MySQL; Docker/Kubernetes). NFR targets are stated independently of the chosen stack.
>
> **Status flags honoured.** Buyer (`DATA-ENT-010`) and PaymentMethod (`DATA-ENT-011`) are `persisted=false`, `status=aspirational/unimplemented` (RC-002); payment capabilities `BIZ-CAP-027/028` are inferred/LOW. Any NFR touching payment data (e.g. PCI-scoped confidentiality) is therefore conditional on those features being implemented and is labelled accordingly.

## 1. Purpose, Scope, and Method

### 1.1 Purpose

This specification defines **measurable, verifiable** non-functional requirements that the forward-engineered system must satisfy. It complements the functional documents (01–11, 13) by constraining *how well* the system behaves rather than *what* it does. Each NFR is expressed as a target value with an objective verification method so that it can be designed for, tested, and accepted.

### 1.2 Scope

Seven NFR categories are specified: **Performance, Availability, Reliability, Scalability, Security, Maintainability, Observability.** Targets apply to the deployable units catalogued in the graph — the storefront `Web` service (`APP-SVC-006`, container `TECH-INF-001`), the `PublicApi` REST service (`APP-SVC-011`, container `TECH-INF-002`), the `BlazorAdmin` SPA (`APP-SVC-016`), and the relational datastore (`TECH-INF-003`; engines `TECH-CUR-020/021`) — under the target deployment topology (containerised, with a microservice-capable decomposition replacing the current monolith-with-cycle `APP-DEP-001`).

### 1.3 Method and column semantics

Each category table uses the columns below.

| Column | Meaning |
|--------|---------|
| **NFR-###** | Forward-engineering NFR identifier (this document). |
| **Requirement** | A single measurable target (p95/p99 latency, RTO/RPO, uptime %, throughput, coverage %, etc.). |
| **Rationale** | Why the target exists, tied to a capability/process/risk. |
| **Source / Basis** | Graph node id(s) and `nfr-registry` NFR-id evidence **where a value exists**; otherwise `DERIVED` or `INDUSTRY-BASELINE` with the governing `ASMP-FE-###`. |
| **Verification method** | How conformance is objectively demonstrated (load test, chaos test, scan, static analysis, audit). |

> **Convention.** `MEASURED` = value present in legacy evidence (`nfr-registry`/graph). `RECOMMENDED BASELINE` = neutral engineering target proposed here (not discovered) and bound to an `ASMP-FE-###`. The legacy evidence overwhelmingly lacks measured targets, so most rows are recommended baselines explicitly labelled as such.
